# Skill: 追踪配置项是否真的被实现

当发现一个配置项"设置了但没生效"时，按此流程确认它是否真的有代码实现。

## 步骤

1. **找到配置的"声明"位置**
   - 环境变量：`.env`、`codeagent.env`、docker-compose 的 `environment`
   - WEBUI 配置页面的显示字段
   - 文档/注释中的配置说明

2. **搜实际代码读取**
   ```bash
   # 搜索整个项目（含依赖库）
   grep -rn "配置变量名" --include="*.py" | grep -v __pycache__
   ```
   - 如果只在**注释**、**文档字符串**、或**WEBUI 展示列表**中出现 → 可疑
   - 如果在业务逻辑中有 `os.environ.get("VAR_NAME")` → 有实现

3. **确认调用链路**
   - 找到读取该变量后的处理逻辑：是赋值、参考、还是仅展示？
   - 关注变量读取后的下游使用：`if auto_continue:` → 外层循环 / WS 续接 / 递归调用

4. **区分"展示"与"执行"**
   - WEBUI 上能看到配置开关 → 只说明前端渲染了它
   - 不等于后端/底层库真正使用了它
   - 典型的"幽灵配置"：frontend display + comment documentation only, no backend consumption

5. **确认后给出结论**
   - 如果确实未实现：明确指出"配置存在于 X 处但无代码读取，功能尚未实现"
   - 如果实现了但未生效：指出调用链路中哪个环节出了问题

## 典型失败模式

| 模式 | 表现 | 根因 |
|------|------|------|
| 幽灵配置 | WEBUI 有开关、文档有说明、但业务代码不读它 | 前端展示和 doc 先行于功能实现 |
| 前缀不匹配 | 用户设 `CODEAGENT_FOO`，库读 `SEED_FOO` | 环境变量名在不同层不一致 |
| 只读不判 | 代码读了变量但赋值后没做任何分支判断 | 半成品/遗留代码 |
