## 一、项目总览

CodeAgent 是一个**自托管的 AI 编码 Agent 平台**，底层基于 Seed 框架。用户通过 WebUI 或 CLI 与 Agent 交互，Agent 自主完成编程、搜索、文件操作等任务。

### 核心能力

- **单 Agent 编码**：全栈开发自主 Agent，覆盖需求→设计→编码→交付全流程
- **多 Agent 团队协作**：Lead Agent 拆任务 → 派发给 Specialist Agent → 合并结果
- **WebUI 可视化**：聊天窗、文件树、团队面板、任务面板、健康看板
- **MCP 扩展**：支持 stdio 和 SSE 传输协议的 MCP 服务器
- **系统自愈**：Watchdog + Supervisor Agent 自动检测/修复异常

### 技术栈

| 层 | 选型 |
|---|------|
| 框架底座 | Seed（自定义 LLM 框架） |
| 后端 | Python + FastAPI（WebUI）+ CLI 交互模式 |
| 前端 | 单页 HTML/JS/CSS（旧）+ React + Vite + TypeScript（v2，待上线） |
| 数据库 | SQLite（`state_store` 持久化） |
| 团队引擎 | 状态机引擎（顺序流水线 / 管家模式 / 广播模式） |

```
└── pyproject.toml
```

### `.codeagent/` 设计原则

1. **物理隔离** — Agent 工作产物与项目源码分开，`.gitignore` 一行搞定
2. **多 Agent 隔离** — 通过 `{agent_id}` 子目录隔离，互不影响
3. **分层规则** — 全局 `rules.md` 所有 Agent 共享，专属 `{agent_id}/rules.md` 仅当前 Agent 遵守
4. **状态不在 system prompt** — `state.md` 由 Agent 自主 `file_read` 读取，system prompt 缓存不受影响
5. **compact 时顺带注入** — 上下文压缩时从 `state.md` 读取最新状态拼入

## 三、三层路径体系

### 第一层：Agent 系统路径

由 Seed 框架在 `build_system_prompt()` 中注入，运行时只读。

| 变量 | 实际路径 | 用途 |
|------|---------|------|
| `$AGENT_HOME` | `~/.codeagent/agents/{id}/` | Agent 主目录 |

| `$AGENT_MEMORY` | `~/.codeagent/agents/{id}/memory/` | 长期记忆目录 |
| `$WORKSPACE` | `Path.cwd()` | 当前工作区根目录 |

来源：`seed.core.config_plane._build_seed_vars_dict()`

### 第二层：项目工作路径

由 CodeAgent 的 `prompt_enrichment._codeagent_vars_dict()` 构建，基于 `Path.cwd() / .codeagent / {agent_id}`。

| 变量 | 路径（相对） | 用途 |
|------|------------|------|
| `$DOCS` | `.codeagent/{id}/docs/` | 需求/设计/任务文档 |
| `$PLANS` | `.codeagent/{id}/plans/` | 工作计划 |
| `$SCRIPTS` | `.codeagent/{id}/scripts/` | 临时脚本 |
| `$TMP` | `.codeagent/{id}/tmp/` | 中间产物 |
| `$SESSION_LOG` | `.codeagent/{id}/sessions/` | 运行日志 |
| `$AGENT_STATE` | `.codeagent/{id}/state.md` | 当前进度 |
| `$SESSIONS_ARTIFACTS` | `~/.codeagent/agents/{id}/sessions/_artifacts/` | 跨会话产物 |

### 第三层：项目代码结构

由 Agent 在 `design.md` / `task.md` 中按「先结构后编码」规则定义，不在本架构文档中硬编码。

## 四、System Prompt 组装流程

### 整体流程

```
fresh_system_prompt()
  │
  ├── seed.build_system_prompt()         ← 读取 agents/{id}/persona/*.md
  │   ├── agent.md                       → render_persona() 展开 $VAR
  │   ├── identity.md                    → render_persona() 展开 $VAR
  │   ├── soul.md                        → render_persona() 展开 $VAR
  │   ├── tools.md                       → render_persona() 展开 $VAR
  │   ├── skills.md                      → render_persona() 展开 $VAR
  │   ├── user.md                        → render_persona() 展开 $VAR
  │   ├── plugin_skill_appendices()      ← 动态技能文件注入
  │   ├── validate_skill_refs()          ← 技能引用验证
  │   └── _path_registry_table()         ← Seed 路径表追加
  │
  ├── fresh_system_prompt() 追加
  │   ├── rules.md                       ← 全局 .codeagent/rules.md
  │   ├── {agent_id}/rules.md            ← Agent 专属 .codeagent/{id}/rules.md
  │   ├── memory.md                      ← render_persona() 展开 $VAR
  │   ├── _codeagent_path_registry()     ← CodeAgent 扩展路径表
  │   ├── _runtime_info_block()          ← 当前时间/OS/工作目录
  │   └── vision_multimodal_appendix()   ← 多模态工具说明
  │
  └── 最终结果
      └── 写入 session.metadata["base_system"]
```

### 缓存机制

```
get_cached_system_prompt()
  │
  ├── _compute_persona_hash()            ← 哈希 persona 文件 + memory.md + rules.md
  │
  ├── 哈希匹配 → 返回缓存 base_system
  └── 哈希不匹配 → 重新 fresh_system_prompt() → 更新缓存
```

- 缓存存于 `session.metadata["base_system"]`
- 哈希存于 `session.metadata["base_system_hash"]`
- 仅首轮 LLM 调用可能重建；后续复用缓存，保持 LLM provider 前缀缓存有效

### 非首轮行为

```python
merge_fresh_system(chat_sess.messages, fresh_sys)
```
- 首轮：替换 `messages[0]` 的 content 为完整 system prompt
- 非首轮：不替换，从缓存读取 → LLM provider 前缀缓存保持命中

## 五、Persona 文件系统

### 文件清单

`~/.codeagent/agents/{agent_id}/persona/` 下存放 6 个 Markdown 文件：

| 文件 | 作用 | 渲染变量 |
|------|------|---------|
| `agent.md` | 工作流程、项目管理规范、上下文管理 | Seed vars |
| `identity.md` | 角色定位、能力范围 | Seed vars |
| `soul.md` | 核心价值观、沟通模式、审美底线 | Seed vars |
| `tools.md` | 工具选用原则、安全规则 | Seed vars |
| `skills.md` | 核心工作流（完整模式/轻量模式） | Seed vars |
| `user.md` | 用户偏好、行为设定 | Seed vars |

### memory.md

独立于 6 文件之外，位于同一目录下：

- 由 `fresh_system_prompt()` 单独读取并追加
- 通过 `render_persona()` 展开 `$VAR`
- 有字符数上限（默认 4000，可配置）
- 内容通过 `self_reflect` 自动积累

### 渲染机制

```python
render_persona(text: str, vars_dict: dict[str, str]) -> str
```

- 白名单替换：只替换 `vars_dict` 中存在的 key
- 支持 `${VAR}` 和 `$VAR` 两种写法
- 支持 `<id>` 占位符填充 agent_id
- 不存在的变量保持原样

### 默认模板

新建用户时，`_ensure_default_persona_files()` 写入基础模板。
可通过 `docs/default-persona-template/` 下的完整模板替换默认内容。

## 六、分层约束系统（rules.md）

### 文件结构

```
.codeagent/
├── rules.md                    ← 项目全局（所有 Agent 共享）
└── agent0/
    └── rules.md                ← Agent0 专属
```

### 加载规则

1. `.codeagent/rules.md` 不存在则跳过
2. `.codeagent/{agent_id}/rules.md` 不存在则跳过
3. 文件内容通过 `fresh_system_prompt()` 追加到 system prompt
4. 文件变化 → persona hash 变化 → system prompt 重建

### 用途

- 大模型缓存策略说明
- 项目特定的编码规范
- 团队协作约定
- API 密钥使用限制
- 不写入 system prompt 的约束说明

## 七、状态管理（state.md）

### 文件位置

`.codeagent/{agent_id}/state.md`

### 写入

Agent 每完成一个 TODO 主动更新：

```python
write_state_file(
    f"wave: W3\n"
    f'todo: "{todo}"\n'
    f"status: completed\n"
    f'next: "{next_todo}"\n',
    agent_id=aid
)
```

### 读取 — 两套机制

**机制 A — 正常运行时**

Agent 自主调 `file_read(.codeagent/{agent_id}/state.md)`，不碰 system prompt 缓存。

**机制 B — compact 发生时**

```
LLMWorker.run()
  → maybe_compact_context_messages()
  → compact 触发
  → _inject_state_into_system(api_msgs, agent_id)
    → 读取 state.md
    → 剥离旧 <<<STATE>>> 块
    → 注入新 <<<STATE>>> 块到 messages[0]
```

compact 后 prompt 结构：

```
messages[0].content:

## File: agent.md
...
## File: identity.md
...

<<<SEED_COMPACT>>>
## 当前项目状态

wave: W3
todo: "[W3.2] 用户登录 API"
status: completed
next: "[W3.3] 前端登录页面"

## Earlier conversation (compressed)
[压缩摘要]
<<<END_SEED_COMPACT>>>
```

## 十、多 Agent 团队系统

### 工作流模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **顺序流水线** | Agent A → B → C 依次执行，A 的输出作为 B 的输入 | 内容处理流水线 |
| **管家模式** | Lead 拆任务 → 并行派发 → 合并结果 | 代码审查、报告生成 |
| **广播模式** | 所有 Agent 收到相同任务，各自输出 | 多角度分析 |

### 核心组件

| 组件 | 功能 |

| `state_store.py` | SQLite 持久化层 |
| `hub.py` | Agent Hub（SSE 消息广播） |
| `self_healing.py` | Watchdog + Supervisor 自愈 |

### 默认 Agent 模板

## 十二、关键数据流

### 一次聊天请求的完整路径

```
用户输入
  │
  ├── CLI: main.py → LLMWorker.run()
  └── WebUI: WebSocket → app_factory → LLMWorker.run()
        │
        ├── get_cached_system_prompt()
        │   ├── 计算 persona hash
        │   ├── 缓存命中 → 返回缓存的 system prompt
        │   └── 缓存未命中 → fresh_system_prompt() → 缓存
        │
        ├── merge_fresh_system() ← 首轮替换 system message
        │
        ├── LLM 请求 → 响应 ← 流式/非流式
        │
        ├── 工具调用 → 执行 → 结果 → 下一轮
        │
        ├── maybe_compact_context_messages()
        │   └── compact 触发 → _inject_state_into_system()
        │
        └── 最终回复 → 返回给用户
```

### 团队运行数据流

```
用户 POST /api/ui/teams/{id}/run
  │
  ├── team_engine.run() 创建 workflow_runs 记录
  │
  ├── 根据模式执行:
  │   ├── 顺序: Agent A → Agent B → Agent C
  │   ├── 管家: Lead 拆 → 并行派发 → 合并
  │   └── 广播: 所有 Agent 并行执行
  │
  ├── 每次 Agent 调用通过 AgentRegistry.call_agent()
  │
  ├── 每步状态写入 state_store
  │
  ├── SSE 实时推送到 WebUI
  │
  ├── Watchdog 每秒扫描心跳
  │
  └── 完成/超时/失败 → 通知用户
```
